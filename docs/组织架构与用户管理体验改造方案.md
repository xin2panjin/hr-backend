# 组织架构与用户管理体验改造方案

## 1. 文档定位

本文是 [用户、角色与权限管理重构方案](用户角色权限管理重构方案.md) 的补充，聚焦“部门树驱动的用户管理体验”。目标是把当前已完成的 IAM/RBAC 能力，组织成更符合管理员日常操作习惯的后台工作台。

本文不重建角色、权限、会话、审计和数据范围模型；这些能力仍以既有 IAM 方案及其 `/iam` 接口为唯一事实来源。本文也不复制旧平台的级联删除组织成员、展示固定明文初始密码等做法。

## 2. 参考系统可借鉴的经验

参考平台将“组织架构”“用户管理”“角色管理”放在同一导航域内：左侧是组织树，右侧是当前组织下的用户清单；用户列表提供角色、账号状态和重置密码等高频操作；角色详情以菜单和功能操作树呈现权限。

其中值得吸收的是信息架构与交互节奏：

1. **组织先于用户**：管理员先定位部门，再在明确的组织上下文中查看和管理人员。
2. **清单承担日常运维**：用户表同时展示人员、主部门、账号、角色和状态，常用操作可就近完成。
3. **角色权限可解释**：权限按模块树分组，避免管理员只看到难以理解的一长串权限码。
4. **高频操作路径短**：新建、编辑、停用、重置密码、查看角色都能从列表或详情抽屉直接进入。

需要明确排除的旧设计包括：删除组织时级联删除全部用户；新建/重置后直接展示固定明文密码；一个用户只能拥有一个角色；父部门自动继承下级业务数据权限。

## 3. 目标与边界

### 3.1 改造目标

1. 建立“左侧部门树 + 右侧用户工作区”的管理页面，减少在部门页和用户页之间切换。
2. 让用户的主所属部门、角色授予、部门数据范围、账号状态和会话状态能够分别查看、分别修改、分别审计。
3. 用可读的模块树解释权限；机器权限码只作为后端契约和研发排障信息，不在管理端页面展示。
4. 所有管理操作继续受后端权限码、数据范围、依赖检查和审计日志保护；页面隐藏不是安全边界。

### 3.2 非目标

1. 不支持部门层级自动带来候选人、职位或简历的数据权限继承。
2. 第一期不开放管理员在线定义未知权限码，也不允许修改内置角色的机器编码；仅允许有权限的管理员在既有权限目录中调整角色的权限勾选关系。
3. 不通过物理删除用户或部门解决“删除”需求。
4. 不把用户主部门等同于招聘专员的负责部门范围。

## 4. 领域规则

### 4.1 部门树与授权范围分离

部门树只表达组织归属与页面筛选上下文：`departments.parent_id` 决定层级，`users.department_id` 是用户唯一的主所属部门。

角色范围仍由 `user_roles` 与 `user_role_scopes` 表达。一个招聘专员可属于“人事部”，但负责“技术部、产品部”的招聘范围；移动其主部门不会隐式修改负责部门范围。

### 4.2 生命周期规则

| 对象 | 页面动作 | 后端语义 |
| --- | --- | --- |
| 部门 | 归档 | 标记为 `ARCHIVED`；存在有效用户、开放职位、有效角色范围或待处理邀请时返回 `409`，先处理依赖 |
| 用户 | 停用 | 标记为 `BLOCKED` 并撤销有效会话；保留角色授予与审计记录 |
| 用户 | 离职 | 标记为 `RESIGNED`，撤销会话，并要求迁移其仍持有的业务责任 |
| 用户 | 删除 | 第一期不提供物理删除入口 |
| 角色授予 | 撤销 | 填写原因，写入审计，并递增目标用户 `authz_version` |

### 4.3 密码与会话规则

管理员不应得到可长期复制的明文密码。邀请用户使用一次性注册链接；管理员重置密码后应使用短时有效的一次性重置凭据或受控临时密码，并要求首次登录修改。任何密码重置、账号停用、角色撤销均必须使该用户旧会话失效。

## 5. 信息架构与页面设计

### 5.1 导航结构

在“权限与安全”下保留三个明确入口：

```text
组织与用户
角色与权限
审计与会话
```

“组织与用户”负责部门和用户的日常运营；“角色与权限”负责查看固定角色、权限解释和角色授予治理；“审计与会话”负责安全追溯。不能以一个巨型页面混合所有职责。

### 5.2 组织与用户工作台

页面使用左树右表布局。

```text
┌───────────────┬──────────────────────────────────────────────────┐
│ 部门树         │ 面包屑：组织 / 人事部 / 招聘组                   │
│ - 搜索部门     │ [用户] [部门详情]                                │
│ - 展开/收起    │ 姓名或邮箱 | 角色 | 状态 | [查询] [重置]         │
│ - 新增根部门   │ [邀请用户]                                       │
│               │ 用户表：姓名、账号、主部门、角色、状态、最后登录、操作 │
│ 选中节点菜单： │ 分页                                              │
│ 新增子部门     │                                                  │
│ 编辑 / 归档    │                                                  │
└───────────────┴──────────────────────────────────────────────────┘
```

设计要求：

1. 默认选中当前登录用户的主所属部门；没有部门时默认根节点或“全部用户”。
2. 选中部门后，用户表默认查询该部门直接成员；“包含下级部门”是显式开关，不能默认扩大结果集。
3. 部门树支持名称/编码搜索、展开收起、空态和归档态标识；归档部门默认隐藏，可由拥有 `department.read` 的用户显式查看。
4. 右键菜单可作为桌面端快捷方式，但新增、编辑、归档必须同时有可发现的按钮，支持键盘操作和移动端降级。
5. 归档前调用依赖摘要接口，面板明确显示待迁移的用户、职位、角色范围和邀请数量，并提供对应处理入口。

### 5.3 用户列表与详情抽屉

用户表建议字段如下：

| 字段 | 说明 |
| --- | --- |
| 姓名 / 邮箱 | 人员识别与唯一账号定位 |
| 主所属部门 | 当前树上下文中的组织归属 |
| 有效角色 | 展示角色名称；悬浮或详情中显示角色编码与有效期 |
| 账号状态 | `ACTIVE`、`BLOCKED`、`RESIGNED`，使用文本和颜色共同表达 |
| 最近登录 | 用于安全运营；无记录时明确显示“从未登录” |
| 操作 | 资料、角色与范围、会话、状态、重置密码，按权限显示 |

点击一条用户后打开详情抽屉，而不是堆叠多个弹窗。抽屉分为四个独立区域：

1. **基础资料**：姓名、手机号、主所属部门；保存需要 `user.update`。
2. **角色与范围**：角色授予记录、授予人、授予时间、有效期、负责部门范围；修改需要 `role.assign`。
3. **账号与会话**：账号状态、登录会话、设备/IP（脱敏展示）、撤销会话；分别需要 `user.disable`、`user.session.revoke`。
4. **审计记录**：只读展示与该用户相关的权限、状态、密码和会话操作；需要 `audit.read`。

每个区域独立提交，避免一次“保存用户”误改资料、部门、角色和安全状态。

### 5.4 部门详情

部门详情显示编码、名称、上级部门、状态、直接用户数、下级部门数、开放职位数和有效角色范围数。部门编码是稳定机器标识，仅在创建时填写；名称与上级部门可调整。

修改上级部门时必须拒绝把节点移动到自身或子孙节点下。该操作只改变组织树，不自动变更职位归属、用户主部门或角色范围；发生跨部门调整时，页面要明确告知这一点。

### 5.5 角色与权限页面

参照参考平台，角色首页使用“列表 + 操作入口”而非在表格中堆叠权限摘要。列表只展示管理员能理解和需要比较的信息：

| 字段 | 说明 |
| --- | --- |
| 角色名称 | 如“系统管理员”“招聘管理员”“招聘专员” |
| 角色说明 | 面向业务管理员的职责描述 |
| 状态 | 启用/停用；系统角色的状态变更需二次确认 |
| 最近修改人 / 时间 | 最近一次角色权限配置变更的操作者与时间 |
| 操作 | `查看权限`、`编辑权限`；按当前用户权限显示 |

角色代码、权限数量、权限码均不在角色列表和权限页面对业务管理员展示。角色代码仍保留在数据库、接口和研发文档中，作为稳定机器标识。

#### 查看权限子页面

从列表的“查看权限”进入只读子页面，例如 `/iam/roles/{role_id}/permissions`。页面以业务模块树展示当前角色可用能力：

```text
用户与组织
  ☑ 查看用户    ☑ 邀请用户    ☑ 编辑用户    ☑ 停用账号
职位管理
  ☑ 查看职位    ☑ 新建职位    ☑ 删除职位
候选人管理
  ☑ 查看候选人  ☑ 更新招聘状态  ☑ 查看 AI 评分
```

每项仅显示中文操作名称和必要的业务说明；页面不可出现 `user.read`、`candidate.update_status` 等机器码。权限项按模块分组，可展开/收起；没有权限的操作使用未勾选状态而不是隐藏，便于理解角色边界。

#### 编辑权限子页面

从列表的“编辑权限”进入独立编辑子页面，例如 `/iam/roles/{role_id}/permissions/edit`。页面沿用同一模块树，但允许勾选和取消勾选既有权限项，提供“模块全选/取消全选”、变更摘要、变更原因和二次确认。

编辑权限不意味着可创建新权限或修改角色机器编码。提交前必须显示“将影响 N 名当前拥有该角色的用户”；保存后写入角色权限审计、递增受影响用户的 `authz_version`，并使其旧访问令牌在下一次请求时失效。该操作仅向拥有 `role.update_permissions` 的高权限管理员开放。

## 6. 接口与数据契约补充

现有 `/iam` 用户、部门、角色、审计和会话接口保持资源边界不变。为支持组织树工作台，建议补充或扩展以下只读契约：

| 接口 | 用途 | 关键约束 |
| --- | --- | --- |
| `GET /iam/departments/tree` | 返回按 `parent_id` 组织的部门树 | 默认不含归档节点；不得附带无权查看的部门信息 |
| `GET /iam/departments/{id}/summary` | 返回直接用户、下级部门、职位、角色范围等摘要 | 与归档依赖摘要区分，摘要不泄露无权业务详情 |
| `GET /iam/users` | 增加 `department_id`、`include_descendants`、`role_code`、`keyword`、`status` 筛选 | `include_descendants` 默认 `false`；分页与排序由服务端执行 |
| `GET /iam/users/{id}/activity` | 返回该用户可见范围内的登录/授权审计摘要 | 需要 `audit.read`，敏感字段脱敏 |
| `GET /iam/roles/{id}/permissions` | 返回角色权限的中文模块树及已勾选状态 | 管理端响应不返回或展示机器权限码 |
| `PUT /iam/roles/{id}/permissions` | 用已定义权限项替换角色权限集合 | 需要 `role.update_permissions`、变更原因与二次确认；未知权限项一律拒绝 |

现有写接口继续承担最终校验：部门创建/编辑/归档、用户资料修改与状态变更、角色授予/撤销和范围替换均不得仅依赖前端树节点或按钮状态。

建议在用户与部门列表响应中增加 `capabilities` 或等价的资源操作能力字段，例如：

```json
{
  "id": "...",
  "status": "ACTIVE",
  "capabilities": ["user.update", "role.assign", "user.session.revoke"]
}
```

它只用于前端体验；服务端仍须在每次写请求重新校验权限和资源范围。

## 7. 后端实现约束

1. 部门树查询必须防环，创建和修改上级部门均需校验不存在自指向或祖先环。
2. 组织树只作为筛选条件时，后端递归查询必须与调用人的部门读取范围相交，不能仅因前端传入 `include_descendants=true` 而返回全量用户。
3. `department.archive` 必须复用既有依赖检查，不得提供绕过依赖的强制归档参数。
4. 用户状态改为 `BLOCKED` 或 `RESIGNED` 时，使用会话服务撤销会话；重新启用不会自动恢复旧会话。
5. 角色、范围、主部门和状态变更分别记录审计动作、操作者、原因、变更前后数据和请求标识。
6. 权限树的展示配置可以由前端维护分组元数据，但权限码合法性仍由后端 `iam/permissions.py` 与数据库字典校验；管理端接口仅下发中文模块、操作名称、说明和勾选状态。
7. 角色权限变更必须在事务内更新 `role_permissions`、记录审计、查询受影响的有效 `user_roles` 并递增这些用户的 `authz_version`；不得只更新角色表而让已登录用户长期保留旧权限。

## 8. 分阶段实施

### 阶段 A：补齐组织树读模型

1. 为部门树、部门摘要和用户组合筛选补充 Repository、Service、Router 与测试。
2. 校验 `parent_id` 的环路、归档节点过滤和下级范围查询。
3. 不改变现有用户、部门、角色的写接口语义。

### 阶段 B：重构“组织与用户”页面

1. 将平铺部门表升级为左树右表工作台。
2. 接入部门上下文、组合筛选、分页、归档预检和用户详情抽屉。
3. 所有按钮使用 principal 权限码和后端 `capabilities` 双重驱动；接口失败显示可行动的错误信息。

### 阶段 C：角色可解释性与安全运营

1. 角色列表提供“查看权限”“编辑权限”操作入口；权限子页面按模块树展示中文操作项，不展示权限码或权限数量摘要。
2. 用户详情接入角色范围、会话撤销和关联审计记录。
3. 增加权限变更后的 principal 刷新、旧会话失效提示与回到登录页的处理。

## 9. 验收标准

1. 管理员可在三层以上部门树中新增、编辑、移动、搜索和归档部门；任何环路操作均被拒绝。
2. 选中部门后仅显示直接成员；开启“包含下级部门”后才合并子部门成员，分页总数正确。
3. 用户主部门变更后不影响其角色范围；角色范围变更后不改变其主部门，两种变更均有独立审计。
4. 有用户、职位、角色范围或邀请依赖的部门无法归档，页面能说明依赖数量并引导处理。
5. 停用、离职、重置密码、撤销角色或会话后，旧访问令牌不能继续访问受保护资源。
6. 没有 `audit.read` 的用户看不到用户审计和角色变更记录；手工调用接口同样被后端拒绝。
7. 角色列表不展示权限摘要或权限码；查看/编辑权限子页面只展示中文模块、操作名称和业务说明，内置角色代码不可被管理端修改。

## 10. 当前实施记录（2026-07-14）

1. 已提供 `GET /iam/departments/tree`、部门工作台摘要和用户按部门（可选包含下级）、角色、状态、关键字的组合筛选。
2. 已将 IAM 前端首页改为“左侧部门树 + 右侧用户工作区”，保留邀请用户、部门归档依赖预检、用户资料修改、角色授予和范围调整。
3. 已增加角色列表及“查看权限”“编辑权限”子页面；页面仅展示中文模块、操作名称和说明，不展示权限码或权限摘要。
4. 已增加 `role.update_permissions` 高风险权限、角色权限替换接口和审计；保存时会递增所有有效角色持有人的 `authz_version`，使旧访问令牌在下一次请求时失效。
5. 创建用户时由管理员填写不可重复的用户名；用户名与用户 ID 绑定且不可编辑，登录入口支持用户名或邮箱。

## 11. 与现有方案的关系

本方案以现有 IAM 重构成果为前提：固定角色、权限码、`user_roles`、`user_role_scopes`、会话撤销、`authz_version`、部门归档依赖检查和审计日志均继续沿用。

后续若扩展自定义角色、跨组织管理、临时授权审批或 HRIS 同步，应在不破坏上述身份与授权边界的前提下单独立项；不要将组织树 UI 的便利性演变为隐式权限继承规则。
