# 候选人沟通与 HR 待办中心产品需求文档

> 文档状态：方案评审稿
> 版本：v1.0
> 更新日期：2026-07-15
> 适用项目：`hr-backend` 与配套候选人端 / HR 工作台前端

## 1. 文档目的

定义“候选人沟通与 HR 待办中心”第一期产品需求，建立从候选人消息到 HR 跟进动作的闭环：

```text
候选人沟通 → AI 结构化洞察 → HR 待办 → 人工处理与流程推进
```

本期以已投递候选人为范围，验证大模型在多轮会话摘要、事实提取、待办识别和证据对齐方面的实际价值。系统不以“自动回复”为目标；所有对候选人的外发消息仍由 HR 人工发送。

本文档用于产品评审、后端与前端拆分、评测集建设和后续销售场景复用。技术实现细节、数据库迁移和接口定义在后续技术设计文档中单独确定。

## 2. 背景与目标

### 2.1 业务背景

当前系统已有候选人、职位、简历、面试和候选人流程能力，但候选人与 HR 的沟通主要分散在邮件或线下渠道。HR 需要反复翻阅历史消息才能判断：

- 候选人正在咨询什么、是否表达了明确意向或异议；
- 是否补充了关键资料、可面试时间或其他承诺；
- 是否存在需要尽快联系、确认或转人工处理的事项；
- 多位候选人中，哪些沟通事项会被遗漏或逾期。

销售场景中的“客户状态识别—任务优先级—会话辅助”与招聘沟通具有相同的底层模式。本期选择招聘作为范围更可控、风险更低的验证场景，沉淀可复用于销售的会话、洞察、证据和待办能力。

### 2.2 建设目标

第一期实现以下目标：

1. 已投递候选人可通过邮箱验证码安全访问自己的投递与会话。
2. 候选人与负责该职位的 HR 可进行双向文本沟通。
3. 每条候选人或 HR 消息进入后，系统异步更新仅供内部使用的结构化会话快照。
4. AI 从会话中识别需要 HR 推进的事项，创建或更新待办并保留原始消息证据。
5. 职位负责人在 HR 工作台集中查看、筛选和处理自己的候选人待办。
6. 建立可评测、可追溯的洞察数据，为后续回复草稿、低风险自动化及销售场景复用打基础。

### 2.3 第一期开箱范围

| 包含 | 不包含 |
| --- | --- |
| 已有投递的候选人邮箱验证码登录 | 候选人自主注册、职位浏览、自主投递、简历上传 |
| 候选人端投递列表、文本会话、消息历史 | 附件、图片、语音、视频等多模态消息 |
| HR 端会话查看与人工回复 | AI 直接回复、AI 回复草稿、FAQ 自动回复 |
| 内部结构化摘要、证据、待办、任务中心 | AI 面试、自动排期、自动状态变更、自动入职办理 |
| 待办在任务中心展示与人工完成 | 站内通知、邮件提醒、企微接入、批量触达 |

## 3. 用户、权限与核心原则

### 3.1 角色

| 角色 | 主要能力 |
| --- | --- |
| 候选人 | 使用邮箱验证码登录；查看本人已有投递及对应会话；发送和查看文本消息。 |
| 职位负责人 / HR | 查看职责范围内的候选人会话；人工回复；查看、处理自己的待办。 |
| 系统管理员 | 沿用现有组织和数据权限，具备管理范围内的查看与审计能力。 |

### 3.2 权限规则

1. 邮箱验证码通过后，仅关联候选人邮箱与已验证邮箱一致的投递记录；一个邮箱可关联多个职位投递。
2. 候选人只能读取和发送属于自己的消息，不能通过候选人 ID、职位 ID 或会话 ID 枚举其他数据。
3. HR 的候选人可见范围继续沿用现有候选人和职位权限策略；LLM 不参与权限判断。
4. 会话摘要、风险标记、置信度、待办和内部建议只对内部 HR 可见，候选人永不看到。
5. 候选人正式状态（如“待面试”“已拒绝”）只能由既有业务流程和人工操作变更；AI 洞察中的沟通阶段不得覆盖正式状态。

### 3.3 产品原则

- **人工优先：** 本期 AI 只理解、提取和提醒，不发送消息、不创建面试、不修改业务状态。
- **证据可追溯：** 每项会话结论和待办必须关联一条或多条原始消息。
- **结构化优先：** 先得到可校验的字段与任务，再渲染供 HR 阅读的摘要，不以自由文本总结替代业务数据。
- **最小数据暴露：** 摘要只记录推进所需的事实，不复制简历全文、联系方式等无关敏感信息。
- **先招聘、后复用：** 核心概念保持可复用，但招聘关联关系必须显式绑定候选人和职位，避免过早抽象成通用 CRM。

## 4. 核心用户流程

### 4.1 候选人登录与进入会话

1. 候选人输入投递时使用的邮箱并获取一次性验证码。
2. 验证成功后，系统查询该邮箱下的已有投递；没有匹配投递时不创建候选人账号或空会话，并提示联系 HR。
3. 候选人进入“我的投递”，选择其中一个职位投递进入对应会话。
4. 候选人可查看该投递的历史消息并发送文本消息。

### 4.2 HR 人工沟通

1. HR 从候选人详情、会话入口或待办中心进入候选人会话。
2. HR 查看候选人消息、当前内部摘要、关联待办及证据消息。
3. HR 人工编辑并发送回复；系统记录消息发送人、发送时间和投递关联。
4. HR 通过既有候选人和面试流程完成状态变更、面试安排等业务动作。

### 4.3 AI 洞察与待办闭环

1. 候选人或 HR 的一条消息成功持久化后，触发异步洞察任务。
2. 任务读取已有快照、最新未处理消息、候选人投递上下文和正式流程状态，输出结构化增量结果。
3. 系统校验输出字段、证据消息归属和置信度；校验失败时不生成待办，记录为待评测或人工复核样本。
4. 通过校验的结果更新会话快照，并按幂等规则创建或更新待办。
5. 职位负责人在“我的候选人待办”中查看任务并处理；任务完成后保留处理结果与来源证据。

## 5. 功能需求

### 5.1 候选人端

| 功能 | 需求 |
| --- | --- |
| 邮箱验证码登录 | 支持获取、校验和过期处理；验证码成功后建立候选人端会话，不使用内部员工账号。 |
| 我的投递 | 展示当前邮箱匹配的已有投递及基础职位信息；候选人按投递进入会话。 |
| 会话历史 | 按时间顺序展示候选人与内部 HR 的可见消息；不展示内部摘要、待办、工具信息或系统推理。 |
| 发送消息 | 首期仅支持非空文本；发送成功后在候选人端立即可见，并进入内部洞察处理链路。 |

### 5.2 HR 会话工作台

| 功能 | 需求 |
| --- | --- |
| 会话查看 | 展示候选人、职位、正式招聘状态、消息历史和内部洞察卡片。 |
| 人工回复 | HR 手工发送文本回复；首期不展示或调用 AI 回复。 |
| 洞察卡片 | 展示当前会话摘要、最新变化、下一步建议、风险提示和置信度；每条结论可跳转或定位到证据消息。 |
| 待办入口 | 展示该候选人/职位下未完成待办，支持进入任务中心处理。 |

### 5.3 结构化会话快照

快照是内部可更新的数据对象。每个“候选人 + 职位投递 + 会话”维护一份当前快照，并保留版本和更新时间。

| 字段 | 说明 |
| --- | --- |
| 当前主题与候选人意图 | 例如政策咨询、薪资咨询、补充资料、面试时间确认、拒绝或其他未知诉求。 |
| 沟通阶段 | 会话层面的沟通状态，如等待 HR 回复、等待候选人补充、待确认时间、已转人工；不等同于正式候选人状态。 |
| 已确认事实 | 仅记录与当前投递推进相关、且有消息证据的事实。 |
| 候选人诉求与承诺 | 候选人的明确问题、可面时间、承诺补充资料等。 |
| 风险与人工处理原因 | 投诉、敏感诉求、信息冲突、低置信或无法安全判断的情况。 |
| 下一步建议 | 面向 HR 的建议，不是自动执行指令。 |
| 证据与置信度 | 每个关键结论至少关联消息 ID；低置信或证据冲突应显式标记。 |

### 5.4 HR 待办中心

| 项目 | 规则 |
| --- | --- |
| 默认负责人 | 该职位的负责人 / 创建人。 |
| 首期待办类型 | 回复候选人、确认面试相关信息、跟进补充资料、人工复核风险或冲突信息。 |
| 优先级 | 高：候选人主动咨询、明确可面时间、投诉/风险；中：明确资料补充或流程确认；低：一般信息补充。 |
| 状态 | 待处理、处理中、已完成、已关闭。 |
| 去重与更新 | 同一候选人、职位和未完成事项应更新既有待办，而非重复创建；新证据或优先级变化应记录。 |
| 首期展示 | 仅在任务中心按负责人、状态、优先级、逾期与职位筛选；不发站内、邮件或外部渠道提醒。 |

## 6. 可复用内核与后续演进

### 6.1 本期沉淀的通用能力

为后续销售场景复用，本期产品和技术设计应稳定沉淀以下概念：

1. **会话：** 某一外部联系对象在具体业务上下文下的沟通容器。
2. **消息：** 标准化的收发消息记录、发送人、来源渠道和时间。
3. **洞察快照：** 可版本化的结构化摘要、状态、事实、风险和置信度。
4. **证据：** 洞察和待办到原始消息的可追溯关联。
5. **待办：** 可分派、可处理、可关闭的人工推进动作。

招聘场景仍须保留“候选人—职位投递”这一强关联；未来销售场景可在同一会话内核上增加“客户—线索 / 商机”关联，而非修改招聘领域规则。

### 6.2 后续阶段

| 阶段 | 扩展内容 | 前置条件 |
| --- | --- | --- |
| 第二期 | 基于制度知识库的回复草稿与 HR 确认发送 | 首期摘要和待办评测达标，知识来源可引用。 |
| 第三期 | 白名单 FAQ、补资料提醒等低风险自动化 | 人工确认数据证明低风险、知识有效期和审计规则完备。 |
| 第四期 | 面试时间协同、AI 面试、入职办理 | 既有排期服务幂等、审批和异常转人工机制完善。 |
| 销售复用 | 企微 / CRM 消息接入、客户状态和销售任务中心 | 会话内核、证据、评测和权限模型经过招聘试点验证。 |

## 7. 评测、验收与风险控制

### 7.1 离线评测

在真实外发或自动化前，建立脱敏标注样本集，至少评测：

- 摘要事实准确率与遗漏率；
- 关键证据消息匹配准确率；
- 候选人意图、沟通阶段和风险标记准确率；
- 待办精确率、召回率、重复待办率；
- 低置信和冲突样本转人工的正确性。

### 7.2 产品验收场景

1. 同一邮箱有多份投递时，候选人只能看到该邮箱对应的全部投递，不可看到其他邮箱候选人数据。
2. 未匹配已有投递的邮箱完成验证码校验后，不能进入候选人端会话或创建新的候选人记录。
3. 候选人和授权 HR 可以在同一投递会话中读写文本消息；未授权 HR 无法访问。
4. 每条消息都能触发内部快照更新；候选人端不显示摘要、风险、待办或证据。
5. 候选人提出明确咨询、可面时间或资料补充承诺时，系统可生成带证据的对应待办，并默认归属职位负责人。
6. 同一未完成事项被后续消息重复提及时，系统更新原待办而不是创建重复任务。
7. AI 洞察失败、低置信或证据不一致时，候选人消息仍正常保存和展示，且系统不会自动发送消息或修改候选人正式状态。

### 7.3 成功指标

首期试点重点观察：

- 有效候选人会话的洞察覆盖率；
- HR 对摘要和待办的人工采纳率；
- HR 处理候选人消息的平均查找时间和待办遗漏率；
- 任务完成率、逾期率与重复待办率；
- 摘要事实、证据和待办质量的离线评测结果。

## 8. 依赖与参考

- 现有候选人、职位、简历、面试及权限体系。
- 现有候选人流程服务：负责正式状态、面试等业务副作用；本期洞察服务只提供建议和待办。
- 现有企业制度知识库：本期不直接对候选人自动回答，后续回复草稿阶段可作为可引用知识源。
- 参考方案：《企微销售对话数据挖掘与 AI 提效应用解决方案》；本期复用其“持续状态识别、任务优先级、人机协作、合规优先、分阶段建设”的原则。
