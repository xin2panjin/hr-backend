# 用户、角色与权限管理重构方案

## 1. 文档目的

将现有的“用户布尔字段 + 路由内判断 + 部门关联”模式重构为可复用的 IAM（Identity and Access Management，身份认证与授权）基础模块。该模块应能作为后续招聘、业务管理和 AI Agent 类系统的权限底座，而不是只适配当前接口。

本文覆盖用户身份、角色、权限、数据范围、会话与审计的设计及迁移路径；不包含前端菜单展示和组织架构树的完整产品设计。

## 2. 当前实现与主要问题

当前用户模型以 `is_superuser`、`is_hr` 和 `managed_departments` 表达身份与范围，权限判断分散在 Router、Repository 和 Service 中。

现状具备 JWT 登录、邀请注册、用户状态控制、部门与 HR 部门绑定、钉钉账号绑定等能力，可以支撑内部开发使用，但不适合作为可演进的权限框架。

主要问题如下：

1. `is_hr`、`is_superuser` 无法表达多个角色、临时授权、角色有效期和角色变更审计。
2. 操作权限与数据范围混在业务代码中，新增接口容易遗漏资源级校验。
3. 候选人列表、人才检索和 AI 评分等入口的数据范围规则不完全一致；同一类数据不能依赖不同的归属字段判断。
4. 后台任务、Agent 工具不一定经过 HTTP Router，只有 Router 依赖无法保证所有调用路径都经过授权。
5. 当前 JWT 没有会话撤销、刷新令牌轮换和令牌唯一标识，用户禁用之外的会话治理能力不足。
6. 钉钉 OAuth 使用用户 ID 作为 `state`，不满足一次性、随机且服务端可核销的 OAuth CSRF 防护要求。

## 3. 目标与边界

### 3.1 建设目标

1. 以标准 RBAC 管理“谁能做什么”。
2. 以数据范围管理“能操作哪些数据”。
3. 让 HTTP Router、Service、后台任务和 Agent 工具共享同一套授权决策。
4. 权限变更、账号状态变更和敏感操作可审计、可撤销、可测试。
5. 保留合理的业务可读性，避免过早构建任意规则引擎或复杂 ABAC 平台。

### 3.2 非目标

第一期不支持管理员在线创建任意权限码、组合任意条件表达式或为单条候选人单独授权。权限码由代码维护；管理员只能在既有角色和权限目录内管理角色权限配置、用户角色授予及其数据范围。

## 4. 设计原则

1. **默认拒绝**：没有角色、权限或数据范围时，不允许访问；“HR 未配置负责部门”必须是空范围，而非全量范围。
2. **职责分离**：认证负责确认调用人身份；授权负责决定操作许可；业务 Service 负责业务规则和副作用。
3. **操作与范围分离**：`candidate.update_status` 是操作权限；“负责部门”或“自己创建的职位”是数据范围。
4. **服务层兜底**：Router 做粗粒度权限门禁，Service/Policy 做资源级校验，禁止只依赖接口层保护。
5. **单一策略来源**：同一资源的 SQL 查询、Milvus 过滤、详情查询和写操作必须由同一个 Policy 产生或复用范围规则。
6. **权限不写入长期 Access Token**：令牌只确认身份与会话；角色、范围及账号状态由服务端实时查询或版本化缓存解析，避免角色变更后权限陈旧。
7. **最小权限与可审计性**：系统管理员应是少量受控角色，所有授权与撤销应记录操作者和原因。

## 5. 目标架构

```text
HTTP Router / Agent Tool / Background Task
                |
                v
      Authentication Dependency
      解析令牌、加载有效用户和会话
                |
                v
      AuthorizationService
      校验权限码，生成数据范围上下文
                |
                v
  CandidatePolicy / PositionPolicy / ResumePolicy
  资源级 can()、ensure()、SQL/Milvus 范围过滤
                |
                v
            业务 Service
```

建议新增以下目录，并保持现有业务目录职责不变：

```text
iam/
  models/                 # 角色、权限、会话、OAuth State、审计模型
  repositories/           # IAM 数据读写
  services/
    authentication_service.py
    authorization_service.py
    token_service.py
    audit_service.py
  policies/
    base.py
    candidate_policy.py
    position_policy.py
    resume_policy.py
  permissions.py          # 权限码常量和默认角色定义
  dependencies.py         # require_permission 等 FastAPI 依赖
```

## 6. 权限模型

### 6.1 角色定义

第一期建议采用固定系统角色，不开放任意自定义角色：

| 角色代码 | 角色说明 | 默认数据范围 |
| --- | --- | --- |
| `ROLE_SYSTEM_ADMIN` | 系统配置、用户与权限治理 | `ALL` |
| `ROLE_HR_ADMIN` | 招聘管理、HR 配置与招聘数据治理 | `ALL` 或按组织配置 |
| `ROLE_HR_RECRUITER` | 招聘专员 | `MANAGED_DEPARTMENTS` |
| `ROLE_HIRING_MANAGER` | 用人部门负责人 | `OWN_POSITIONS` |
| `ROLE_EMPLOYEE` | 普通员工 | `SELF` |

`ROLE_SYSTEM_ADMIN` 不应被当作普通业务角色使用。日常招聘管理优先授予 `ROLE_HR_ADMIN` 或 `ROLE_HR_RECRUITER`。

角色码是固定的全大写机器标识，格式为 `ROLE_*`；部门码采用 `DEPT-业务域-序号`，例如 `DEPT-HR-001`、`DEPT-ENGINEERING-001`。部门码规范化为全大写且不可使用自由格式；开发期历史数据直接编号为 `DEPT-000001`，不保留 `legacy-*` 等临时代码。

### 6.2 权限码

权限码以“资源.动作”命名，在 `iam/permissions.py` 中定义并通过迁移/种子同步到数据库。例如：

```text
user.read                 user.manage
role.read                 role.assign                 role.update_permissions
position.read             position.create             position.delete
candidate.read            candidate.create            candidate.update_status
candidate.read_ai_score   resume.upload               resume.parse
talent_search.query       assistant.use
```

权限码是代码契约，数据库中的 `permissions` 用于关联、审计和查询；不允许管理端录入未知权限码。

其中用户与部门管理不使用笼统的 `user.manage` 作为唯一权限，应至少拆分为：

```text
user.read                 user.invite                 user.update
user.disable              user.reset_password         user.session.revoke
department.read           department.create           department.update
department.archive
```

这样可以让系统管理员、人事管理员和只读审计人员拥有不同的管理边界；高风险操作（禁用账号、重置密码、归档部门、撤销会话）也能被准确审计。

### 6.3 数据范围

数据范围使用固定枚举：

| 范围 | 含义 |
| --- | --- |
| `ALL` | 全部数据 |
| `MANAGED_DEPARTMENTS` | 当前角色被授予的部门集合 |
| `OWN_POSITIONS` | 当前用户创建的职位及其关联候选人 |
| `SELF` | 当前用户自身拥有的资源 |
| `NONE` | 无数据权限，默认拒绝 |

多个角色拥有同一权限时，采用允许范围的并集；不引入显式拒绝规则。账号被禁用、会话被撤销或资源状态不满足业务规则时，始终拒绝。

### 6.4 资源归属约定

候选人的访问范围统一以其 `position` 为准：

- 部门范围：`candidate.position.department_id`；
- 用人负责人范围：`candidate.position.creator_id`；
- 个人资源范围：简历以 `resume.uploader_id` 为准。

禁止在某个入口使用 `candidate.creator_id`、另一个入口使用 `position.creator_id` 来表达同一“可见候选人”规则。候选人创建人仅作为业务追溯字段，不作为默认访问边界。

## 7. 数据模型

### 7.1 RBAC 基础表

```text
roles
  id, code(unique), name, description, is_system, created_at, updated_at

permissions
  id, code(unique), name, resource, action, description

role_permissions
  role_id, permission_id

user_roles
  id, user_id, role_id, assigned_by, assigned_at, expires_at, revoked_at, revoke_reason
```

`user_roles` 使用独立主键，便于将某次角色授予与其数据范围、审计记录关联。有效角色的唯一性应通过部分唯一索引或服务层约束保证。

### 7.2 角色数据范围表

```text
user_role_scopes
  id, user_role_id, scope_type, department_id(nullable), created_at
```

第一期仅支持部门范围：`scope_type=DEPARTMENT` 且填写 `department_id`。`ALL`、`OWN_POSITIONS`、`SELF` 由角色权限的预定义范围推导，无需为每个用户重复存储。

现有 `hr_managed_departments` 迁移到该表后删除，避免“部门已分配但用户不是 HR”这类不一致状态。

### 7.3 会话与 OAuth 表

```text
auth_sessions
  id, user_id, refresh_token_hash, jti, issued_at, expires_at,
  revoked_at, revoke_reason, replaced_by_session_id, device_info, ip_address

oauth_states
  id, provider, state_hash, user_id, redirect_uri,
  expires_at, consumed_at, created_at
```

刷新令牌只存哈希；OAuth `state` 只存哈希并在回调成功或失败后立即核销。

### 7.4 审计表

```text
audit_logs
  id, actor_id, action, target_type, target_id,
  before_data, after_data, request_id, ip_address, created_at
```

至少记录角色授予/撤销、范围变更、账号禁用/启用、会话撤销、敏感候选人状态变更和 AI 评分读取。

### 7.5 用户与部门的生命周期字段

用户和部门均属于组织主数据，不应以物理删除实现“删除”功能；否则会破坏职位、候选人、角色范围、会话和审计记录的外键关联与追溯能力。

```text
departments
  在现有字段上新增：code(unique), status(ACTIVE/ARCHIVED),
  parent_id(nullable), archived_at, archived_by

users
  保留：department_id（主所属部门）、status(ACTIVE/BLOCKED/RESIGNED)
  新增：authz_version, last_login_at, disabled_at, disabled_by
```

第一期可以只使用平级部门，但保留 `parent_id` 以支持后续组织树；父子层级不自动继承候选人或招聘数据权限，是否继承必须在未来单独定义。部门的 `code` 为稳定机器标识，名称可修改；`code`、有效部门名称均应具备唯一性约束。归档后部门不可再被选为用户主部门、职位部门或角色范围。

## 8. 策略层设计

Policy 是资源权限的唯一入口。以候选人为例，应至少提供：

```python
class CandidatePolicy:
    async def can_read(self, actor, candidate) -> bool: ...
    async def ensure_can_update_status(self, actor, candidate) -> None: ...
    def build_sql_scope(self, actor): ...
    def build_milvus_filter(self, actor): ...
```

调用规则如下：

1. `require_permission()` 在 Router 入口检查操作权限。
2. `CandidateService` 加载目标候选人后调用 `ensure_can_update_status()`；后台任务或 Agent 直接调用 Service 时同样生效。
3. 候选人列表、详情、AI 评分查询通过 `build_sql_scope()` 过滤。
4. 人才搜索先使用 `build_milvus_filter()` 限缩召回集合，再使用同一 Policy 生成的 PostgreSQL 条件二次复核。

写操作除了授权，还必须校验业务状态机。例如拥有 `candidate.update_status` 不代表可以跳过候选人状态流转规则。

## 9. 接口与依赖边界

### 9.1 FastAPI 依赖

建议提供：

```python
require_permission(Permission.CANDIDATE_READ)
require_permission(Permission.CANDIDATE_UPDATE_STATUS)
get_current_principal()
```

`get_current_principal()` 返回已验证的用户、有效角色和授权版本，而不是将 `is_hr`、`is_superuser` 暴露给业务代码。

### 9.2 IAM 管理接口

保留现有 `/user` 登录和个人信息接口以降低前端迁移成本；管理类接口统一迁移至 `/iam`。用户、部门、角色和范围分别是独立资源，不能继续复用“给 HR 分配部门”的专用接口表达所有管理行为。

```text
# 当前主体与用户管理
GET    /iam/me
GET    /iam/users                         # 分页、部门、状态、角色筛选
GET    /iam/users/{user_id}
POST   /iam/invitations                   # 邀请创建用户，不直接写入密码
PATCH  /iam/users/{user_id}               # 姓名、手机号、主所属部门等基础资料
PATCH  /iam/users/{user_id}/status        # ACTIVE / BLOCKED / RESIGNED
POST   /iam/users/{user_id}/reset-password
POST   /iam/users/{user_id}/sessions/revoke

# 部门管理
GET    /iam/departments                   # 默认仅返回 ACTIVE，可显式查询 ARCHIVED
GET    /iam/departments/{department_id}
POST   /iam/departments
PATCH  /iam/departments/{department_id}   # code、名称、描述、parent_id
DELETE /iam/departments/{department_id}   # 语义为归档，不物理删除

# 角色、权限与数据范围
GET    /iam/roles
GET    /iam/permissions
GET    /iam/roles/{role_id}/permissions
PUT    /iam/roles/{role_id}/permissions
POST   /iam/users/{user_id}/roles
DELETE /iam/user-roles/{user_role_id}
PUT    /iam/user-roles/{user_role_id}/scopes/departments
```

接口命名中的 `DELETE /iam/departments/{department_id}` 只是 REST 操作语义，实际执行归档。用户不提供物理 `DELETE`：离职使用 `RESIGNED`，临时停用使用 `BLOCKED`。邀请用户和角色/范围授权必须在同一事务内完成，并写入审计日志。

### 9.3 用户与部门 CRUD 业务规则

**部门创建与修改**

1. 创建部门时校验 `code`、有效名称唯一，父部门存在且为 `ACTIVE`，并记录创建审计。
2. 修改部门时，禁止把部门移动到自身或子孙节点；第一期未启用层级时，接口仍校验 `parent_id` 为 `null`。
3. 修改部门名称或描述不会改变已有职位、候选人和角色范围的归属；这些关系均保持 `department_id` 不变。

**部门归档（删除）**

1. 若部门仍有 `ACTIVE`/`BLOCKED` 用户、未关闭职位、有效 `user_role_scopes` 或待处理邀请，则返回 `409 Conflict` 并给出各类依赖数量。
2. 管理员需先完成用户转部门、职位迁移/关闭、范围撤销和邀请取消，才可归档部门。
3. 归档后保留历史关联、审计和查询能力，但不再出现在默认下拉选项及新建表单中。

**用户创建、修改与停用**

1. 管理员通过邀请创建账号，邀请必须指定主所属部门、初始角色、按角色要求的部门范围和失效时间；注册完成后原子创建用户、角色和范围。
2. 用户基础资料修改与角色/范围修改拆分为不同接口和权限，避免一次普通资料编辑隐式提高权限。
3. 转部门需校验目标部门为 `ACTIVE`；它只改变用户主所属部门，不自动扩大或缩小其 `ROLE_HR_RECRUITER` 的负责部门范围，范围变更必须显式操作并审计。
4. 禁用用户时撤销其全部有效会话；离职用户标记为 `RESIGNED` 并撤销角色与会话。两者均保留数据和审计记录。
5. 不允许通过任意更新接口修改 `is_hr`、`is_superuser`、密码哈希、授权版本等内部字段；密码重置必须走专用令牌或管理员重置流程。

## 10. 认证与会话改造

1. Access Token 使用短有效期，建议 15 至 30 分钟；Refresh Token 生命周期由 `auth_sessions` 管理。
2. JWT 使用标准声明：`sub`（用户 ID）、`jti`、`iat`、`nbf`、`exp`、`iss`、`aud`、`typ`。用户 ID 不再放在 `iss`，令牌类型不再复用 `sub`。
3. 刷新接口执行 Refresh Token Rotation：旧会话撤销，签发新会话；检测到已替换令牌重复使用时撤销该用户全部会话。
4. 用户禁用、密码修改、角色/范围变更时增加 `authz_version` 或撤销相关会话，确保权限及时生效。
5. 登录错误对外统一提示，内部记录失败次数、IP 和原因；增加限流与渐进式锁定策略。
6. 邀请码改为持久化邀请记录，保存哈希、失效时间、使用时间、目标角色和部门范围，禁止仅使用短数字验证码作为唯一凭证。
7. 钉钉 OAuth 发起时创建随机 state 并持久化；回调时校验 provider、有效期、未消费状态和绑定用户，随后核销。若钉钉协议支持，启用 PKCE。

## 11. 分阶段迁移计划

### 阶段一：规则收敛与越权修复

目标：不改现有角色表结构，先建立 Policy 接口和测试基线。

1. 定义权限码、角色矩阵和候选人归属规则。
2. 新增 `AuthorizationService`、`CandidatePolicy`、`PositionPolicy`、`ResumePolicy`。
3. 修复职位列表/删除、候选人创建与状态更新、AI 评分读取、简历解析任务查询、人才搜索的资源级校验。
4. 明确 HR 空部门范围为拒绝/空结果，并让 SQL 与 Milvus 一致。
5. 添加越权测试和 Agent 工具调用测试。
6. 候选人检索索引中的普通用户归属字段切换为职位创建人后，执行一次全量回灌，确保已有 Milvus 数据与 PostgreSQL 策略一致：`uv run python -m scripts.rebuild_candidate_milvus_index`。

### 阶段二：RBAC 数据库迁移

目标：引入新表并双读验证。

1. 创建 `roles`、`permissions`、`role_permissions`、`user_roles`、`user_role_scopes`、`audit_logs` 迁移，并为 `departments`、`users` 补齐生命周期字段和索引。
2. 种子化固定角色和权限码。
3. 回填用户角色：`is_superuser=True` 回填 `ROLE_SYSTEM_ADMIN`；`is_hr=True` 回填 `ROLE_HR_RECRUITER`；现有普通业务用户由业务确认后回填 `ROLE_HIRING_MANAGER` 或 `ROLE_EMPLOYEE`。
4. 将 `hr_managed_departments` 回填为对应 `user_role_scopes`。
5. 双读比对旧、新授权结果；出现差异时记录日志，不静默放大权限。
6. 将现有部门补齐稳定 `code`，将历史用户的 `department_id`、状态和角色授予关系做完整性校验；不满足归档约束的部门不得进入删除流程。

### 阶段三：切换与清理

目标：新模型成为唯一授权来源。

1. 所有 Router、Service、Agent 和检索过滤切到 Policy。
2. 新增 IAM 管理接口和审计查询能力。
3. 删除 `is_hr`、`hr_managed_departments` 及业务代码中的直接布尔判断。
4. `is_superuser` 仅在过渡期保留；确认无调用后迁移至 `ROLE_SYSTEM_ADMIN` 并删除字段。

### 阶段四：认证强化

目标：完成会话、OAuth 与账号安全治理。

1. 上线会话表、刷新轮换、注销和会话撤销。
2. 上线随机一次性 OAuth state 与回调核销。
3. 上线登录限流、密码策略、邀请记录和安全审计告警。

### 当前实施记录（2026-07-14）

1. 阶段一已完成：候选人、职位、简历与人才搜索已收敛至基于有效 RBAC 角色与范围的 Policy，并补充越权测试；Milvus 历史索引仍需在发布窗口执行一次全量回灌。

7. 已提供短生命周期 Access Token、带 `sub/sid/jti/ver/typ/iss/aud` 的 JWT、Refresh Token Rotation、重复使用检测后的全会话撤销、管理员重置密码和用户单会话撤销接口。
8. 已提供 `GET /iam/audit-logs`，支持按操作者、动作、目标和时间分页筛选用户、部门、角色、邀请与会话产生的审计记录；它由 `audit.read` 权限码保护。
9. IAM、业务 Router、Service、人才搜索与招聘助手均以有效 RBAC 角色和范围作为唯一授权来源；`is_hr`、`is_superuser`、`hr_managed_departments`、旧员工/HR 管理接口和双读服务均已删除。
10. 钉钉授权已改为持久化、随机且一次性的 OAuth state；回调核销后才继续交换授权码。当前钉钉 OAuth 协议接入未启用 PKCE，需在确认服务端支持对应参数后再补充。
11. 配套前端已接入 `/iam/me` principal Store、权限码菜单/路由守卫和 IAM 管理页：支持邀请用户、用户资料/主部门/状态修改、部门新增/编辑/归档依赖预检、角色授予/撤销及招聘专员范围调整；当前用户自身权限变化后会刷新 principal 并收敛菜单与路由。
12. 前端请求层已接入一次性并发合并的 Refresh Token Rotation：受保护请求收到 `401` 时使用 Refresh Token 重试一次，刷新失败则清空本地状态并带原始地址跳转登录；登录成功后安全地回跳原页面。Refresh Token 目前仍因后端接口契约暂存于 `localStorage`，待 Cookie 化后移除。
13. 角色和部门编码已正式化：角色改为固定 `ROLE_*` 值，部门使用 `DEPT-*` 格式；迁移 `j0e1f2a3b4c5` 会直接改写开发数据，不保留旧码别名或 `legacy-*` 回填值。
14. 已上线统一密码策略：注册与管理员重置均要求 12 至 128 位，且同时包含大写、小写、数字和特殊字符；禁止空白字符、用户名/邮箱前缀和常见弱口令。按当前决定，登录限流暂不实施。
15. 已新增部门树与用户组合筛选接口，前端 IAM 首页已改为“部门树 + 用户工作区”；角色列表不展示权限摘要或机器权限码，改为提供“查看权限”“编辑权限”子页面。编辑权限由 `role.update_permissions` 保护，变更会审计并使所有受影响用户的授权版本失效。

## 12. 测试与验收标准

### 12.1 权限矩阵测试

对每个角色和每个关键资源建立参数化测试，至少覆盖：

- 系统管理员读取与管理全部资源；
- 招聘专员仅可访问其负责部门的职位、候选人、AI 评分和人才搜索结果；
- 用人负责人仅可管理本人创建职位的候选人；
- 普通员工不能创建职位、查看候选人或调用招聘助手；
- 空部门范围 HR 无法通过 SQL、Milvus、详情或 Agent 工具获得候选人数据；
- 非法用户不能读取他人的简历解析任务和 AI 评分；
- 角色撤销、账号禁用后，已有会话立即失去相应权限。

### 12.2 数据一致性测试

同一调用人、同一候选人在候选人列表、候选人详情、AI 评分、人才检索 PostgreSQL 复核及 Agent 工具中的允许/拒绝结果必须一致。

### 12.3 安全验收

1. Access Token 不包含角色、部门或可长期复用的权限快照。
2. Refresh Token 可撤销、轮换且只保存哈希。
3. OAuth `state` 不可预测、一次性、过期后失效。
4. 所有角色和范围变更均有审计日志。
5. 所有敏感接口都有认证、操作权限和资源范围三层检查。

## 13. 前端页面与交互改造

### 13.1 前端改造原则

前端负责展示当前用户可用的功能、引导用户完成角色和范围配置，并对无权限状态提供明确反馈；它不是授权边界。所有接口仍必须由后端完成认证、操作权限和资源范围校验。

因此，前端不得继续以 `user.is_hr`、`user.is_superuser` 作为业务判断来源，也不得因为按钮隐藏而假设接口安全。应统一使用后端下发的权限码进行 UI 判断，并将角色、范围变更后的权限刷新纳入应用状态管理。

### 13.2 当前前端涉及的改造点

现有 Vue/Vite 前端已具备用户 Store、路由、用户 API、员工管理页面和主布局。权限改造会影响以下现有入口：

| 现有入口 | 改造内容 |
| --- | --- |
| 登录页与用户 Store | 登录后获取当前主体（principal）与权限快照；刷新、注销和登录失效时统一清理状态 |
| `MainLayout` 导航菜单 | 依据权限码控制菜单项可见性；不可把菜单隐藏作为后端鉴权替代品 |
| 员工管理页 | 从“状态管理 + 部门筛选”扩展为用户、角色、部门数据范围的管理入口 |
| 职位、候选人、人才搜索页 | 按权限决定操作按钮与敏感字段展示；列表空态须区分“无数据”和“无负责范围” |
| 招聘助手页 | 无 `assistant.use` 时不展示入口；后端返回 403 时应提供无权限提示，不应只显示请求失败 |

### 13.3 当前主体与权限状态

应用启动、登录成功、刷新令牌成功、角色或范围变更后，都应请求 `GET /iam/me` 并更新用户 Store。建议返回的前端契约如下：

```json
{
  "user": {
    "id": "...",
    "realname": "...",
    "status": "ACTIVE",
    "department": {"id": "...", "name": "技术部"}
  },
  "roles": ["ROLE_HIRING_MANAGER"],
  "permissions": ["position.read", "position.create", "candidate.read"],
  "authz_version": 3
}
```

说明：

1. `permissions` 用于菜单、按钮、路由守卫和页面展示；角色名称仅用于身份说明，不应散落在业务组件的 if/else 中。
2. 不向普通用户下发其他部门、其他用户的范围配置；仅在其有管理权限的页面下发本次表单必需的部门选项。
3. 当接口返回“权限版本已变化”、401 或刷新失败时，前端清空 principal 和本地 UI 缓存，回到登录页；不能继续沿用旧的 `localStorage.user` 判断权限。
4. Access Token 仅用于请求认证。若前后端部署条件允许，Refresh Token 应迁移为 `HttpOnly + Secure + SameSite` Cookie，前端不读取、不持久化 Refresh Token。

建议新增 `src/stores/principal.ts`（或将职责明确地并入现有 `user` Store）、`src/composables/usePermission.ts` 与 `src/directives/permission.ts`：

```ts
const { can, canAny, refreshPrincipal } = usePermission()

can('candidate.update_status')
// 模板示例：<el-button v-permission="'position.create'">新建职位</el-button>
```

组件只调用 `can(permissionCode)`，不直接判断角色代码或部门 ID。路由可在 `meta.permissions` 中声明页面所需权限，由全局守卫提前拦截；进入页面后的资源级权限仍以接口响应为准。

### 13.4 页面信息架构

建议在系统设置下新增“权限与安全”分组，仅向 `user.manage`、`role.read` 或 `role.assign` 的用户显示：

| 页面 | 主要内容 | 关键交互与限制 |
| --- | --- | --- |
| 用户管理 | 用户列表、账号状态、所属部门、已授予角色 | 支持筛选、邀请、查看、编辑基础资料、转部门、禁用/离职；角色以标签显示，禁止编辑布尔角色字段 |
| 部门管理 | 部门编码、名称、描述、父部门、在职人数、职位数、有效范围数 | 支持新增、编辑、归档；归档前展示依赖检查和迁移清单，不能直接物理删除 |
| 用户角色与范围抽屉 | 角色、有效期、负责部门、授权原因 | 选择 `ROLE_HR_RECRUITER` 时必须选择至少一个负责部门；`ROLE_SYSTEM_ADMIN` 需要二次确认和填写原因 |
| 角色目录 | 角色名称、角色说明、状态、最近修改人/时间，以及“查看权限”“编辑权限”操作入口 | 列表不展示权限摘要或权限码；权限以中文模块树在独立查看/编辑子页面呈现。只能勾选既有权限项，不能在线新建未知权限码或修改角色机器编码；编辑权限需要 `role.update_permissions`、原因、二次确认、审计和受影响用户权限版本失效 |
| 部门范围配置 | 已授予的部门范围及其生效角色 | 仅展示当前待编辑用户可配置的部门；保存后提示该用户需要重新获取权限状态 |
| 我的账号与会话 | 个人资料、钉钉绑定状态、当前/其他登录会话 | 支持注销当前会话、撤销其他会话和重新绑定钉钉 |
| 审计日志 | 角色、范围、账号状态、会话与敏感操作记录 | 仅向审计权限用户开放；支持按操作者、目标用户、动作和时间筛选 |

邀请用户页也需要同步调整：邀请表单除邮箱和所属部门外，增加“初始角色”“负责部门范围（按角色需要展示）”“授权说明”。前端不允许提交不完整的角色范围组合；后端仍必须再次校验并将邀请、角色和范围在注册完成时原子落库。用户编辑页应将“基础资料、账号状态、角色与范围、会话”拆为独立区域和提交动作，避免误操作。

### 13.5 业务页面的权限交互

1. **职位管理**：只有 `position.create` 才显示“新建职位”；删除按钮还应基于后端返回的资源能力或当前权限显示。不要按“同部门”在前端自行推断可删除性。
2. **候选人列表与详情**：列表、详情、AI 评分、状态流转分别使用独立权限码。没有候选人读取权限时不显示入口；资源范围不足时后端应返回不泄露资源存在性的结果，前端展示统一的“无权访问或记录不存在”。
3. **人才搜索与招聘助手**：检索结果只能展示后端过滤后的候选人。前端不得根据用户选择的部门拼接可信访问条件；筛选项仅是查询条件，不是权限条件。
4. **空范围状态**：HR 拥有招聘功能但尚未分配负责部门时，页面显示“尚未配置负责部门，请联系管理员”，而不是误导为“暂无候选人”。
5. **权限变更即时性**：管理员完成角色或范围变更后，刷新被编辑用户的权限版本；当前管理员自身权限变更时，立即刷新 principal、菜单和当前路由。若当前页面已不可访问，跳转到首页并提示权限已更新。
6. **部门归档**：部门详情页先请求依赖摘要并展示“需转移的用户、职位、角色范围、邀请数量”；存在依赖时禁用归档确认，提供跳转到相应处理页面的入口。

### 13.6 请求、错误处理与路由守卫

统一在请求层处理以下情况：

| 响应 | 前端行为 |
| --- | --- |
| 401 | 尝试一次刷新会话；失败则清空状态并跳转登录页，保留原目标地址以便登录后返回 |
| 403 | 保留当前页面上下文，展示“无此操作权限”；不要自动当作登录失效 |
| 404（受保护资源） | 展示“记录不存在或无权访问”，避免枚举资源是否存在 |
| 权限版本冲突/失效 | 重新拉取 `/iam/me`；仍无权限则收敛菜单并跳转到可访问页面 |

全局路由守卫只负责页面级体验，例如 `/settings/iam` 需要 `role.read`。按钮级控制使用 `usePermission()`；对象级可操作性由后端在列表/详情响应中返回，或由操作接口最终裁决。前端不得缓存某次“允许”结果后无限期复用。

### 13.7 前后端联调与迁移顺序

1. 后端先提供兼容的 `GET /iam/me` 与权限码字典，旧登录响应中的 `is_hr`、`is_superuser` 暂不删除。
2. 前端接入 principal Store、请求拦截、路由守卫和 `usePermission()`，并将现有页面中的布尔判断逐步替换为权限码判断。
3. 增加权限与安全管理页面；角色/范围修改成功后调用 `refreshPrincipal()`，并刷新被编辑用户的列表状态。
4. 后端完成 RBAC 切换、所有业务接口完成资源级授权后，前端移除旧布尔字段依赖与兼容分支。
5. 最后切换 Refresh Token Cookie、会话管理页和 OAuth 绑定页，完成安全能力联调。

### 13.8 前端验收标准

1. 不同角色登录后，菜单、路由、按钮和敏感字段展示符合权限矩阵。
2. 手动构造或重放前端请求不能绕过后端权限校验；前端隐藏按钮不作为验收结论。
3. HR 无部门范围时，候选人、人才搜索与招聘助手均不会显示其他部门数据，并展示明确引导。
4. 管理员变更当前用户角色/范围后，无需重新打开浏览器即可看到菜单和可操作项收敛或扩展。
5. 401、403、404、会话刷新失败、角色变更后的过期页面都有统一且可理解的提示。
6. 角色管理表单对角色、部门范围、有效期和高风险管理员授权具备前端校验、二次确认和加载/失败态。
7. 用户邀请、资料编辑、转部门、禁用/离职、部门新增/编辑/归档均有完整加载、冲突、失败和成功后的列表刷新状态；归档操作不会造成历史数据在页面中丢失。

## 14. 需要业务确认的决策

在实施数据库迁移前，需要确认以下规则：

1. 普通员工是否允许创建职位，还是只有 `ROLE_HIRING_MANAGER` 可创建？
2. 用人负责人能否删除同部门其他人创建的职位，还是仅限职位创建人？
3. `ROLE_HR_ADMIN` 是否拥有全公司候选人数据，还是也必须指定部门范围？
4. 候选人创建人是否有独立于职位创建人的额外查看权限？
5. 是否需要“面试官”角色，以及其可见范围是仅本人面试安排还是候选人完整资料？
6. 一个用户是否始终只有一个主所属部门；若未来支持兼职部门，哪些业务数据以主部门为准？
7. 部门归档时，职位是必须全部迁移/关闭，还是允许连同部门进入历史归档状态？

这些决策会影响默认角色矩阵，但不改变本文的 RBAC + 数据范围总体架构。
