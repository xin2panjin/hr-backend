# 知识库多策略文本切分改造方案

## 1. 目标与边界

在不改变现有“文本提取 → 清洗 → 预览 → 配置快照 → 异步索引 → Embedding → Milvus”链路的前提下，为知识库增加可选的正文切分策略。管理员可在上传前预览不同策略的切片效果，并将选择固化到索引任务中。

本期支持四种策略：

| 策略值 | 名称 | 实现方式 |
| --- | --- | --- |
| `structured_builtin` | 句子聚合 | 保留现有按章节/页码分组、句末标点切分、超长字符兜底的实现 |
| `fixed_length` | 固定长度 | 使用 LangChain `CharacterTextSplitter`，以空分隔符按固定长度和重叠切分 |
| `custom_character` | 固定字符切分 | 使用 LangChain `CharacterTextSplitter`，由管理员指定一个分隔字符或字符串 |
| `langchain_recursive` | 递归分段 | 使用 LangChain `RecursiveCharacterTextSplitter`，默认提供中文友好分隔符且允许管理员调整顺序 |

不改变的部分：PDF/DOCX/Markdown/TXT 提取、章节路径、PDF 页码、清洗规则、切片持久化字段、来源引用、Milvus Schema、Embedding 调用和权限模型。

## 2. 设计原则

1. 所有策略均先按 `section_path + page_number` 分组，再切分正文；每个子切片继承该组的章节与页码。
2. 预览和实际索引均通过同一个策略工厂创建 splitter，确保结果一致。
3. 每个索引任务在 `task_metadata.processing_config` 中保存 `strategy`、自定义分隔符与长度配置；重试和重建不依赖页面当前默认值。
4. LangChain 只负责正文分段，不接管文件解析、清洗、存储、索引或业务权限。
5. 对现有历史任务，缺失 `strategy` 时自动回退为 `structured_builtin`，保持兼容。

## 3. 配置契约

```json
{
  "cleaning": {"normalize_whitespace": true},
  "chunking": {
    "strategy": "custom_character",
    "max_characters": 500,
    "overlap_characters": 80,
    "custom_separator": "。"
  }
}
```

- `custom_separator` 仅在 `custom_character` 策略生效，必须为 1 至 20 个字符的非空字符串。
- `recursive_separators` 仅在 `langchain_recursive` 策略生效；按数组顺序递归降级，空字符串只能作为最终字符级兜底。
- 选择 `fixed_length`、`custom_character`、`langchain_recursive` 时，服务端必须依赖 `langchain-text-splitters`；依赖不存在时明确报错，不可悄悄切换成另一策略。

## 6. 验收标准

- 四种策略均可通过预览 API 返回切片，且切片保留原章节路径和页码。
- 同一文件、同一配置在预览和索引前生成相同正文序列。
- 自定义分隔符配置非法时返回 400；非自定义策略不受该字段影响。
- 未携带策略的历史任务继续使用 `structured_builtin`。
- 前端只在“自定义分隔符”模式下展示分隔符输入，并在修改后提示重新预览。

## 7. D3 对比实验使用方式

同一制度文档应以四种策略分别“预览 → 上传/重建索引 → 运行 D2 评测脚本”，并保存每次输出的 JSON/Markdown 报告。每次实验需记录：策略值、最大长度、重叠长度、自定义分隔符、文档版本、Embedding 模型、检索配置和 Reranker 配置。

建议固定 `max_characters=500`、`overlap_characters=80` 作为第一轮基线，仅切换策略；确认策略差异后，再单独比较长度、重叠和分隔符优先级。重建索引会替换当前文档的旧切片，因此若需保留多组实验结果，应为每组创建独立的测试文档版本或独立知识库 Collection。
